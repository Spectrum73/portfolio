Summary

This is a personal project I've used to learn about Unity and game creation, I have also used it to learn about Blender for 3D asset creation, including how to export and implement them in-engine. The game itself is an action platformer, featuring fast, fluid movement and combo-oriented combat.

Player Controller

The player controller features a number of movement options including: wallrunning, dashing and swinging from a grappling hook. It uses the Kinematic Character Controller Asset for its physics and collision handling.

Combat

Combat uses traditional fighting game combo principles, featuring cancels, specials and other systems such as an action buffer is used to improve responsiveness. Most systems were implemented from scratch so that I could learn how they should work in more detail whilst making them.


Attacks for the combat system use a modular approach, being constructed from multiple smaller actions playing at set times throughout the attack, some of these actions include: Activating a hitbox, playing VFX or changing the actor's velocity.

Similarly, hitboxes are also modular, instances can easily have their hit data adjusted, including the number and placement of hitboxes.


Due to ScriptableObjects not being MonoBehaviours it is less suitable to check their hits using colliders or triggers. Instead, Physics.OverlapSphereNonAlloc is used which doesn't generate any garbage, this is especially important as it is likely to be called frequently.

Interfaces are used for hit and hurt responders so objects can decide how they should react and validate hits.







Other

Most systems are made modular for reusability, usually via dependency injection with interfaces. An example for this is the Conditions system which allows specific conditions to be evaluated.

Any class wanting to use conditions only needs to reference either the Condition class for a single condition or create its own ConditionEvaluator which will handle multiple conditions for it. Conditions also support events for when they become true or false to help avoid the need to poll them.

By using these, complex condition classes can exist without the script referencing requiring to know anything other than the base class. Two examples are buttons, which merely evaluate when they're pressed or not, and an EnemyGroup, which evaluates itself each time one of its registered enemies dies.
